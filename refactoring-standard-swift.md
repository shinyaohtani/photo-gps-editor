# Swift リファクタリング規準書

以下のコードを、Swiftらしく（Swifty）、かつ堅牢で読みやすいコードになるようにリファクタリングしてください。
伺いは必要ありません。まずコードを提案してください。

## 1. 基本方針

- 既存のコードを、Swiftの慣習に沿ったスマートかつ読みやすい形（Swiftyなコード）にリファクタリングすることを目的とする。
- コード全体の構造（型定義や拡張）を見ただけで、データと処理の流れが把握できるように整理する。

## 2. リファクタリング手順

- コード全体の構成を理解したうえで、
- 必要に応じて **Struct / Class / Enum** への分割、責務の明確化を行い、**振る舞いは原則としてインスタンスメソッドに配置**し、
- 処理の簡素化（`map`, `filter`, `reduce` や `guard` の活用）や統合を実施してスマートな構成に変換し、
- 代替の全コードを提示し、コードのフローや設計意図を簡単に説明し、
- 曖昧な点がある場合は確認しつつ、基本的には意図を汲み取り進めてください。
- 全ての指示に合致しているか確認し、合致していない部分が見つかったら最初からやり直してください。
- コード実行結果（副作用含む）が変更されてしまう場合、リファクタリング失敗であるため、やり直してください。

## 3. モノ指向型設計（Object/Value-Oriented）

- オブジェクト指向および値指向（Value Semantics）を基礎概念として扱い、主体を型（Struct/Class）にし、少しでも動作を示唆するものは型として定義しないでください。
- 動詞や "-er"、"-or"、動詞にもなり得る言葉で終わるような名称は動作を示唆しているため、型名としては適切ではありません。（例: `MailFormatter` NG -> `MailFormat` Good）
- 型にするものは `Asset` や `Result`、`Structure` など「モノ（名詞）」に特化し、`Runner` や `Translator`, `Matcher`, `Exporter` など機能（Functionality）を型名にしないでください。
- メソッド名は「そのオブジェクトが何をするか（Action）」または「そのオブジェクトの何か（Property）」を表す視点で命名してください。
- `get` プレフィックスはSwiftの命名規約（API Design Guidelines）に反するため禁止します。
    - 値を取得する場合は **Computed Property** （計算プロパティ）として定義するか、引数が必要な場合のみ動詞から始まるメソッドにしてください。
    - 例: `XXXAuth` 型の `url` を取得する場合 → `var url: URL { get }`
- 型定義（Struct/Class）が多くなることは問題ではありません。
- 型名を見て「何を表すデータ/オブジェクトなのか」、全体はどのような流れなのかが理解できるようにしてください。
- すべてのメソッドは型（またはその `extension`）に含めてください。Global functionとして定義しないでください。
- `static func` / `class func` は**原則禁止**です。振る舞いは、責務を持つインスタンス（`struct` / `class`）へ委譲してください。型メソッドに本体ロジックを書かないでください。
- 例外は、**Appleフレームワークやプロトコル適合で型メソッドが必須**な場合、または**真に不変な定数公開のための `static let`** のみとします。この場合も、`static func` / `class func` は薄い入口に留め、本体ロジックは別の責務を持つ型へ委譲してください。

## 4. メソッド・構造制約

- 1つの型（または主要な `extension` ブロック）に含まれるメソッドは4〜5個を目安にしてください。3個未満では少なすぎ、6個以上は多すぎます（責務過多の兆候です）。必須です。
    - **注意: メソッド数は同一型のすべての `extension` を合算してカウントしてください。** `extension` に分割しても総数は変わりません。責務が多い場合は型自体の分割を検討してください。

### カウント対象の定義

- **カウント対象**: インスタンスメソッド（`func`）、型メソッド（`static func`, `class func`）、複数行の `var` computed property
- **カウント対象外**:
    - `init`, `required init?(coder:)`, `deinit`（初期化子/終了子）
    - 1行の computed property（`var x: T { y }` 形式のゲッター）

### Apple定義プロトコル準拠メソッドの除外

Apple定義プロトコル（`UICollectionViewDataSource`, `UICollectionViewDelegateFlowLayout`, `UIScrollViewDelegate`, `MKMapViewDelegate`, `UIContextMenuInteractionDelegate` 等）の準拠メソッドは、メソッド数カウントから**除外**する。カウント対象は型が独自に定義したメソッドのみとする。

理由:
- シグネチャがApple固定であり変更不可能
- 省略するとコンパイルエラーとなり削除不可能
- `self` が必要であり別型に移動すると委譲コストが増す

ただし、プロトコル準拠メソッドが多すぎる場合（8個以上）は、DataSource/Delegate を別オブジェクトに分離することを検討する。

### 型メソッドの扱い

- `static func` / `class func` は、オブジェクトへの権限委譲ができていない兆候として扱い、**原則として採用しない**。
- 型メソッドが必要に見える場合は、まず「その振る舞いを持つべき具体的なオブジェクト」を新設できないか検討すること。
- 生成・変換・判定・検索などを型メソッドで表現したくなった場合も、まずは対応する名詞型へ責務を移し、そのインスタンスメソッドとして表現すること。
- 例外は以下のみ:
    1. Apple定義プロトコルやライブラリ制約により、シグネチャとして型メソッドが必須
    2. `static let` による真に不変な定数公開
- 例外 1 の場合でも、型メソッド内部に本体ロジックを書かず、委譲だけを行うこと。

### 単一責務型の最小メソッド数

単一プロトコル準拠や単一責務の型は、メソッド数が3未満でも許容する。他の型と統合して責務が混在するよりは、少数メソッドの型を維持すること。

例: `UIContextMenuInteractionDelegate` のみを責務とする型が1メソッドであることは適切。無理に他の型と統合して「ジェスチャ＋コンテキストメニュー」のような混在責務にしない。

- メソッドは必ず25行以下に短くしてください。必須です。
    - `guard let` や `defer` を活用してネストを浅くし、行数を抑えてください。
    - 26行以上になる場合は `private` なヘルパーメソッドに分割してください。
- アクセス修飾子（`private`, `fileprivate`, `internal`, `public`）を適切に使用し、外部に公開する必要のないメソッドやプロパティは徹底して隠蔽してください。
    - Pythonの `_` 接頭辞ルールは廃止し、Swiftの言語機能としてのアクセス制御を使用してください。
- すべて型を明記してください。型推論に頼りすぎず、メソッドの引数・返り値は明示的に記述してください。
    - `Optional` は `Type?` を使用し、適切にアンラップ（`if let`, `guard let`）またはオプショナルチェーン扱ってください。
- クラスプロパティ（`static let`）とインスタンスプロパティを明確に区別してください。
- `static let` は、真に不変で共有してよい定数に限定してください。設定値・依存オブジェクト・状態の置き場として使わないでください。
- `static var` は原則禁止です。共有可変状態は責務分離を壊しやすいため、専用の状態オブジェクトへ移してください。
- `static func` / `class func` がやむを得ず存在する場合でも、分岐・集計・状態遷移・依存生成を持たせず、委譲だけを行う薄い入口にしてください。
- `self` はクロージャ内でのキャプチャ明示や、プロパティ名と引数名の衝突回避など、コンパイラが要求する場合以外は省略を推奨します（Swiftの標準的なスタイル）。

## 5. 命名規約

- メソッド名、変数名、型名は、Swift API Design Guidelinesに従い、明瞭かつ役割が明確になるように命名してください。
- 抽象的で意味をなさない単語（`info`, `data`, `manager`, `process`, `asset`）は変数名や型名に使わないでください。文脈を含んだ具体的な名前にしてください。
- Pathの扱いは `String` ではなく `URL` 型を使用することを前提とし、変数名は `absoluteURL` や `relativeURL` のように明記してください。
- ファイル名は `fileName`、ディレクトリ名は `directoryName` とし、略語（`fname` 等）は使用せず CamelCase で記述してください。
- 配列（Array）や辞書（Dictionary）であることを示すために `List` や `Dict` という接尾辞をつけるのは禁止です。
    - 複数形（例: `items`, `users`）を使用することでコレクションであることを表現してください。
- **Lower Camel Case**（先頭小文字）を変数・メソッド名に使用し、**Upper Camel Case**（先頭大文字）を型名（Struct/Class/Enum）に使用してください。
    - Pythonの Snake case (`total_failures`) はSwiftでは非推奨です。`totalFailures` としてください。
- 極力1単語で表現し、どうしても誤解を招く場合のみ単語を連結してください。

## 6. 差分チェック対応

- リファクタリング前後で、標準出力・エラー出力・ファイル出力などの挙動（Behavior）が完全に一致するようにしてください。
- Swiftはコンパイル言語ですが、ロジックの等価性を保つため、リファクタリング後のコードが元のコードと同じ目的・同じ結果を返すことを論理的に保証してください。

## 7. スタイルガイド・Lint適合

- 生成されるコードはコンパイル可能であるだけでなく、**SwiftLint** のデフォルトルールに適合する記述にしてください。
- **swift-format** などの自動フォーマッターを通した際に、大幅な差分が発生しない標準的なスタイルを遵守してください。
    - インデントはスペース4つを使用してください。
    - コロン(`:`)の後ろにはスペースを入れ、前には入れないでください（例: `variable: Type`）。
    - 演算子の前後には適切なスペースを入れてください。
- 強制アンラップ (`!`) や `try!` は、Lintで警告対象（Force Cast/Force Try）となるため、原則禁止とします（テストコードを除く）。
    - 代わりに `guard let`, `if let`, `do-catch`, `Optional Chaining` を使用して安全に処理してください。
- 垂直方向の空白（Vertical Whitespace）を適切に入れ、メソッド間や論理ブロックの間に1行の空行を挟んで可読性を確保してください。

### SwiftLint と SwiftFormat の併用

- **SwiftLint** と **swift-format** の両方で違反が0件となる状態を維持してください。
- swift-format を適用すると行数が増える場合があります（長い引数リストの改行、guard文の展開など）。その場合でも本規準書のルール（メソッド25行以下等）を満たす必要があります。
- 両ツールの競合が発生した場合の対処法：
    1. swift-format を先に適用する
    2. SwiftLint 違反が発生した場合、手動で修正する
    3. 関数が長くなった場合は、ヘルパー関数や状態構造体（State Struct）への分割で対応する
- セミコロン（`;`）による複文は swift-format で禁止されるため、必ず改行してください。
- Optional の初期化で `= nil` は冗長です。`var prev: Point?` のように省略してください。

## 8. その他

- Path操作には `String` 操作や `NSString` を使わず、すべて標準の **`URL` 構造体** および **`FileManager`** を使用してください。
- 非同期処理が含まれる場合は、Completion Handlerではなく、可能な限り Swift Concurrency (`async/await`) を使用して可読性を高めてください。

### SwiftUI View の型分割

- SwiftUI View の `@State` / `@Binding` プロパティに紐づくロジックは、`@Observable` class に移動することで型分割可能とする。
- `extension` によるファイル分割ではメソッド数は減らない（全 extension 合算ルール）。`@Observable` class への移動を推奨する。
- 子 View（`DedupTopBar`, `DedupFooter` 等）への分離は、値とクロージャを引数として渡す形式で行い、`@Binding` の多用を避ける。

```swift
// 推奨パターン
@Observable @MainActor
final class DedupModel {
    var cur: Int
    var selected: [Int: Set<String>]
    // state + ロジック（4-5メソッド）
}

struct DedupView: View {
    @State private var model: DedupModel
    // 純粋なUI組立のみ（4-5メソッド）
}
```

## 9. 簡潔な命名（2単語ルール）

- 型名は **2単語以内** で表現してください。3単語以上の結合は冗長であり、可読性を損ないます。
    - 例: `BridgeEdgePointPosition` NG → `BridgePos` Good
    - 例: `OriginalEdgePointPosition` NG → `EdgePos` Good
    - 例: `BridgeTraceContext` NG → `TraceScope` Good
    - 例: `OuterBoundaryResult` NG → `OuterResult` Good
- 文脈から明らかな場合、修飾語を省略してください。
    - 例: `BridgePointKind` → `PointKind`（Bridge文脈内なら `Point` で十分）
- Enum の case 名も簡潔に。
    - 例: `case bridgeEndpoint(seqIndex: Int)` → `case bridge(seqIdx: Int)`

## 10. 変数名の短縮ルール

頻出する変数名は、文脈が明確な場合に限り短縮形を使用してください。ただし、意味が不明瞭になる略語は避けてください。

### 推奨する短縮形

| 完全形 | 短縮形 | 備考 |
|--------|--------|------|
| `sequence` | `seq` | |
| `position` | `pos` | 型名・変数名両方 |
| `coordinate` | `coord` | |
| `candidates` | `cands` | |
| `previous` | `prev` | |
| `current` | `cur` | |
| `index` | `idx` | |
| `result` | なし | 短縮しない |

### 避けるべき短縮形

| 完全形 | 避けるべき短縮形 | 理由 |
|--------|------------------|------|
| `crossings` | `xs` | 日本人には直感的でない |
| `bridge` | `br` | 意味が不明瞭 |
| `vertex` | `v` | 1文字は避ける（ループ変数除く） |

### 適用ルール

- ローカル変数やプライベートメソッドの引数には積極的に短縮形を使用してください。
- 公開API（`public` / `internal`）では完全形を維持し、可読性を優先してください。
- 同一スコープ内では一貫した命名を使用してください（`seq` と `sequence` を混在させない）。
